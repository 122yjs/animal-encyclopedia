# 동물도감 54 — 종합 개선계획 보고서

> 작성일: 2026-04-24
> 대상: 초등 3학년 과학 "동물의 생활" 디지털 도감
> 기준 버전: 정적 HTML/CSS/JS (零 npm runtime dep)

---

## 1. 개요

본 보고서는 실제 수업 현장 운용 피드백과 코드베이스 UX 감사 결과를 종합하여, **학습 동기 유발**, **사용성**, **안정성**, **몰입도** 4개 축으로 개선 방향을 제시합니다.

## 2. 문제 정의 및 개선 목표

| 축 | 현재 상태 | 목표 |
|---|---|---|
| **학습 동기** | 54마리를 한꺼번에 수집 → 부담/지루함 | 지역별 마일스톤 + 보상으로 지속적 동기 부여 |
| **사용성** | 시작 화면 정보 과부하, 모달 버튼 가려짐, 상태바 비가시 | 초등생에게 맞는 단계적 정보 공개 + 핵심 UI 항시 가시 |
| **안정성** | Wikimedia 외부 이미지 의존, localStorage 미처리 | 오프라인/차단 환경에서도 핵심 콘텐츠 제공 |
| **몰입도** | 무음, 단조로운 피드백 | 포획/완성 시 다중 감각 피드백(시각+청각) |

## 3. 상세 개선 항목

### 3.1 온보딩(튜토리얼) 도입
**요청**: "너무 많은 정보가 한 번에 나오는 시작화면 → 동물 퀴즈 하나를 예시로 시작"

**문제 분석**
- 현재 첫 화면에 사이드바(모드/묶음/설정) + 안전 약속 + 진행도 + 검색 + 54개 카드가 동시에 노출
- 초등 저학년(8~9세)은 동시에 5개 이상의 정보 요소를 처리하기 어려움
- "도감 모드"와 "분류 미션"의 차이, "등록"의 의미가 직관적이지 않음

**개선안**
1. **첫 방문 시 온보팅 오버레이** 3단계
   - ① "동물 카드를 눌러 사진과 설명을 봐요" (카드 하나 하이라이트)
   - ② "퀴즈를 풀어 도감에 등록해요" (퀴즈 버튼 하이라이트 + 데모 퀴즈 1문제)
   - ③ "모든 동물을 모아 도감 마스터가 되어 보세요!" (진행도 하이라이트)
2. **"다시 보지 않음"** 체크 또는 `localStorage`로 이후 방문 시 스킵
3. **안전 약속은 접었다 펼치기**로 변경하여 시작 화면 복잡도 완화

**영향 파일**: `index.html`, `no-question.html`, `app.js`, `styles.css`
**예상 난이도**: 중간

### 3.2 지역별(묶음별) 마일스톤 & 보상 시스템
**요청**: "54마리 모두 하기엔 너무 많고 지루함 → 지역별로 한 번씩 도감 완성 → 지역 넘어가는 식으로"

**문제 분석**
- 현재 진행도는 "0 / 54" 전체 집계만 제공
- 54마리는 초등 3학년 1차시(40분) 내에 완료하기엔 과다
- 중간 성취감(보상)이 없어 지속 몰입이 어려움

**개선안**
1. **5개 지역(묶음)별 독립 진행도**
   - 우리 주변(11) / 땅(13) / 물(16) / 특별한 환경(9) / 전체(54)
   - 각 지역별 "X / Y" 진행도를 사이드바 필터 버튼 옆에 미니 배지로 표시
2. **지역 완성 시 보상 오버레이**
   - "🎉 우리 주변 도감 완성!" + 다음 지역 안내 버튼
   - 연출: 황금색 별 효과 + 축하 메시지
3. **전체 완성 시 최종 보상 페이지**
   - "🏆 도감 마스터 달성!" + 수집한 동물 썸네일 콜라주
   - "다시 보기" / "처음부터 다시 시작" 선택지
   - 캡처/공유 유도 (선택적)
4. **진행도 상태바 항시 표시**
   - 현재 스크롤 시 진행도가 사라지는 구조
   - 모바일: 상단 고정 미니 상태바 ("🐾 12 / 54")
   - 데스크톱: 사이드바에 항상 노출 (현재는 사이드바에 있으나 게임 모드 시 사라짐)
   - **게임 모드에서도 최소 진행도 노출** (ex: 상단바)

**영향 파일**: `app.js`, `styles.css`, `index.html`, `no-question.html`
**예상 난이도**: 높음 (상태 구조 변경 + 애니메이션)

### 3.3 모달/퀴즈 버튼 가시성 개선 (가로·태블릿)
**요청**: "가로-태블릿에서 모달창 퀴즈 버튼이 한 번에 안 보임"

**문제 분석**
- 현재 `.detail-sheet`는 `max-height: 90vh`, `overflow-y: auto`
- 퀴즈 영역(`.quiz-box`)이 본문 하단에 위치하여, 작은 화면/가로 모드에서는 스크롤 없이 보이지 않음
- 초등생은 "아래로 스크롤하면 더 있다"는 사실을 인지하지 못하는 경우가 많음

**개선안**
1. **퀴즈 버튼 영역 상단 고정(Anchor)**
   - 퀴즈 시작 시 "🎯 퀴즈 풀고 도감에 등록하기" 버튼을 상단(`.modal-title-row` 아래)으로 이동
   - 또는 시작 버튼을 누르면 자동으로 퀴즈 영역으로 `scrollIntoView({ block: 'start' })`
2. **가로 모드(landscape) 최적화**
   - `@media (orientation: landscape) and (max-height: 600px)`에서
     - `.detail-sheet`를 `flex-direction: row`로 변경 (사진 왼쪽 / 정보 오른쪽)
     - 퀴즈 버튼이 항상 우측 하단에 고정
3. **스크롤 힌트**
   - 모달 하단에 미니 그라데이션 + "아래로 내려보세요 👇" 3초간 표시

**영향 파일**: `styles.css`, `app.js`
**예상 난이도**: 중간

### 3.4 외부 링크 돌아가기 UX
**요청**: "잘못 링크 눌렀을 때 돌아가기 링크 더 명시(위키백과 등)"

**문제 분석**
- 상세 모달 하단의 "사진 출처 보기"는 위키백과로 새 탭 이동
- 아이들이 실수로 클릭하면 도감으로 돌아오는 방법이 불분명 (브라우저 뒤로가기에 의존)
- 모바일 브라우저에서는 뒤로가기 버튼이 잘 보이지 않음

**개선안**
1. **외부 링크 아이콘 + 경고 추가**
   - `source-link` 옆에 "🌐 새 창에서 열어요" 문구 추가
   - 클릭 시 간단 툴팁: "위키백과를 보고 싶다면 [계속], 도감으로 돌아가려면 이 창을 닫아요"
2. **iframe 프리뷰 대체 (선택적)**
   - 위키백과 링크를 직접 열지 않고, 모달 내 작은 "출처 미리보기" 패널로 축약 정보 제공
   - "더 보기" 클릭 시에만 외부 이동
3. **no-question 버전에서도 출처 링크가 있다면 동일 적용**

**영향 파일**: `app.js`, `styles.css`
**예상 난이도**: 낮음

### 3.5 효과음 시스템
**요청**: "효과음 등"

**문제 분석**
- 현재 모든 인터랙션이 무음
- 포켓볼 포획 애니메이션은 시각적으로 훌륭하나 청각 피드백이 없어 몰입도가 절반에 그침
- 학교 현장에서 소리가 나오지 않는 환경(헤드폰 미사용)도 고려해야 함

**개선안**
1. **핵심 사운드 4종 내장 (Web Audio API 또는 `<audio>`)**
   - `sounds/catch-start.mp3` — 포켓볼 던질 때
   - `sounds/catch-success.mp3` — 포획 성공 시
   - `sounds/quiz-correct.mp3` — 퀴즈 정답
   - `sounds/quiz-wrong.mp3` — 퀴즈 오답
   - `sounds/region-complete.mp3` — 지역 완성
2. **음소거/켜기 토글 버튼**
   - 사이드바 하단 또는 모달 상단에 🔊/🔇 버튼
   - `localStorage`로 사용자 설정 유지
3. **접근성 고려**
   - `prefers-reduced-motion`과 마찬가지로 `prefers-reduced-motion`이 아닌 별도 설정
   - 사운드가 없는 환경에서도 시각적 피드백이 100% 유지되어야 함
4. **용량 제약**
   - 각 사운드 30KB 이하, 총 150KB 이하로 제한 (정적 사이트 특성상)
   - Web Audio API의 Oscillator로 간단한 합성음(피카츄 스타일이 아닌 단순 효과음)도 고려 가능

**영향 파일**: `app.js`, `index.html`, `no-question.html`, `styles.css` + `sounds/` 디렉토리 신규
**예상 난이도**: 중간

### 3.6 이미지 내장 및 오프라인 대응
**요청**: "특정 환경에서 연결된 Wiki 이미지가 로딩되지 않음 → 용량 허용 범위 내에서 최대한 고화질 이미지 파일 내장"

**문제 분석**
- 현재 54마리 이미지를 모두 Wikimedia 외부 URL에 의존
- 학교 네트워크 방화벽, 느린 연결, 위키미디어 차단 시 모든 카드가 "😿 사진을 불러오지 못했어요"
- `getImageSource()`의 fetch 실패 시 사용자 경험이 급격히 저하
- 빌드 시점에 이미지를 prefetch하지 않음

**개선안**
1. **이미지 내장 전략**
   - `images/` 디렉토리에 54마리 썸네일(카드용) + 상세(모달용) 저장
   - 카드 썸네일: 200x150px, JPEG 60% 품질, 목표 5~8KB/장 × 54 = 270~430KB
   - 모달 상세: 600x450px, JPEG 70% 품질, 목표 20~30KB/장 × 54 = 1.0~1.6MB
   - 총 1.5MB 이하로 제한 (GitHub Pages/정적 호스팅 부담 고려)
2. **빌드 스크립트 개선**
   - `scripts/build.js`에 `images/` → `dist/images/` 복사 단계 추가
   - `imageSources` 객체를 빌드 시점에 `images/고양이.jpg` 등 로컬 경로로 오버라이드
   - 개발 환경에서는 여전히 Wikimedia 사용 가능 (빠른 로드)
3. **폴백 체인**
   - `local image` → `Wikimedia API` → `SVG placeholder`
   - 내장 이미지가 먼저 로드되고, 실패 시에만 외부 fetch 시도
4. **리포지토리 용량 관리**
   - Git LFS 고려 또는
   - 빌드 시 외부 다운로드 + `.gitignore` 처리 (빌드 머신에서만 이미지 확보)

**영향 파일**: `scripts/build.js`, `app.js`, `imageSources` 데이터 구조
**예상 난이도**: 높음 (용량/품질 트레이드오프 + 빌드 파이프라인 변경)

### 3.7 기타 기술적 개선 (감사 결과 반영)

#### 3.7.1 검색 디바운스
- 현재 `input` 이벤트마다 전체 카드 재렌더링
- 150ms 디바운스 적용으로 입력 중 불필요한 렌더링 제거
- **난이도**: 낮음

#### 3.7.2 localStorage 쓰기 안정성
- `saveCollected()`, `saveQuestionToolUrl()` 등에 `try/catch` 추가
- 저장 실패 시 사용자 알림: "진행 기록을 저장할 수 없어요. 사생활 보호 모드가 켜져 있나요?"
- **난이도**: 낮음

#### 3.7.3 `no-question.html` 빌드 자동화
- 현재 `index.html`과 수동 동기화 → 실수 위험
- `scripts/build.js`에서 `index.html`을 기반으로 질문방 기능만 제거하여 자동 생성
- **난이도**: 중간

#### 3.7.4 게임 모드 모바일 최적화
- 게임 토큰 크기 확대 (현재 78~90px → 최소 100px)
- "카드 탭 → 영역 탭" 2단계 플로우를 모바일 기본으로 변경
- **난이도**: 중간

#### 3.7.5 `inert` 폴리필 (선택적)
- 구형 iPad(iOS 14 이하)에서 모달 포커스 트랩 미작동 가능성
- `inert` polyfill 2KB 추가 또는 `aria-hidden` fallback
- **난이도**: 낮음

## 5. 구현 시 주의사항

### 5.1 정적 사이트 제약
- 런타임 npm 의존성이 없으므로, 모든 기능은 Vanilla JS로 구현
- Web Audio API는 대부분의 모던 브라우저에서 지원되나 iOS Safari에서는 사용자 제스처 이후에만 재생 가능 → 온보딩 "시작하기" 버튼 클릭 이후 초기화

### 5.2 용량 예산
- 현재 전체 리포지토리(코드): ~50KB
- 이미지 내장 시: +1.5MB (최대)
- 효과음: +150KB
- **총 배포 용량 목표**: 2MB 이하 (GitHub Pages 로드 타임 3초 이내 유지)

### 5.3 두 HTML 파일 동기화

## 7. 결론

현재 동물도감 54는 **초등학생 대상 디지털 학습 콘텐츠로서 시각적 완성도와 접근성이 이미 높은 수준**입니다. 포켓볼 포획 애니메이션, 퀴즈-힌트-재도전 사이클, 교사-학생 워크플로우 분리 등은 교육 현장에서의 가치가 입증된 기능입니다.

**가장 시급한 개선**은 **이미지 안정성(3.6)**과 **지역별 마일스톤(3.2)**입니다. 외부 이미지 의존성을 제거하고, 54마리라는 큰 목표를 4~5개의 작은 목표로 쪼개는 것만으로도 실제 수업 현장에서의 완성도와 만족도가 크게 향상될 것입니다.

그 다음으로 **온보딩(3.1)**과 **퀴즈 버튼 가시성(3.3)**을 개선하면, 처음 접하는 학생들의 진입 장벽을 낮추고 태블릿 환경에서의 사용성을 확보할 수 있습니다.

본 보고서의 P0~P1 항목을 순차적으로 적용한 후, P2 효과음과 P3 폴리필을 마무리하면 **교실에서 안정적으로 운용할 수 있는 완성도 높은 교육 콘텐츠**가 될 것입니다.
